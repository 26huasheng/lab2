### 2.2 接口详细设计

#### 2.2.1 IFileSystem — 文件系统抽象接口（Strategy 模式）

```java
public interface IFileSystem {
    List<String> readLines(String path);              // 读取文件所有行
    void writeLines(String path, List<String> lines);  // 写入文件 (覆盖)
    void appendLine(String path, String line);         // 追加一行
    boolean exists(String path);                       // 判断文件/目录存在
    List<String> listDirectory(String path);            // 列出目录内容
}
```

**设计要点：**
- 所有业务代码通过此接口进行依赖注入，禁止直接使用 `java.io.File`
- 两个实现类：[LocalFileSystem](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/core/LocalFileSystem.java)（真实文件操作）和 [MockFileSystem](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/core/MockFileSystem.java)（内存模拟，用于 TDD 测试）
- 所有 `IOException` 统一包装为 `EditorException` 抛出

#### 2.2.2 IEditor — 文本编辑器核心接口

```java
public interface IEditor {
    void append(String text);                                          // 末尾追加文本
    void insert(int line, int col, String text);                       // 指定位置插入
    String delete(int line, int col, int len);                         // 删除指定长度字符
    String replace(int line, int col, int len, String text);           // 替换文本
    Iterator<String> getLineIterator(int startLine, int endLine);      // 行迭代器
    boolean isModified();                                              // 是否被修改
    void setModified(boolean modified);                                // 设置修改状态
    String getFilePath();                                              // 获取文件路径
}
```

**设计要点：**
- 所有行号和列号**均从 1 开始计数**
- `replace` 内部实现为：先 `delete` 再 `insert`
- `getLineIterator` 返回 `Iterator<String>`，隐藏内部 `List` 实现细节
- `delete` 返回被删除的子串，供 Command 模式撤销时使用

#### 2.2.3 ICommand — 命令接口

```java
public interface ICommand {
    void execute();              // 执行命令
    void undo();                 // 撤销命令
    String getCommandLog();      // 获取日志字符串
}
```

**设计要点：**
- 命令具备完整的**可撤销**语义
- `getCommandLog()` 返回格式化的日志字符串，格式如：`20251024 09:41:40 append "text"`

#### 2.2.4 ITreeNode — 树节点接口

```java
public interface ITreeNode {
    String getName();                  // 获取节点名称
    List<ITreeNode> getChildren();     // 获取子节点列表
}
```

**设计要点：**
- 最简单的树形结构接口，仅定义名称和子节点两个核心操作
- 配合 [TreePrinter](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/cli/TreePrinter.java) 实现目录树的递归渲染

#### 2.2.5 ISubject & ICommandObserver — 观察者模式接口

```java
public interface ISubject {
    void attachObserver(ICommandObserver o);    // 注册观察者
    void detachObserver(ICommandObserver o);    // 移除观察者
    void notifyObservers(String msg);            // 通知所有观察者
}

public interface ICommandObserver {
    void onCommandExecuted(String logMessage);   // 命令执行回调
}
```

**设计要点：**
- 完全解耦了**命令执行**和**日志记录**两个关注点
- Subject 不关心 Observer 的具体实现，Observer 不关心命令细节

## 三、设计模式详解

### 3.1 Command 模式（命令模式）

**模式目的：** 将请求封装为对象，使系统支持命令的**参数化**、**排队**、**日志化**以及**撤销/重做**。

**参与角色：**

| 角色 | 对应类 |
|------|--------|
| **Command**（抽象命令） | [ICommand](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/command/ICommand.java) |
| **ConcreteCommand**（具体命令） | [AppendCommand](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/command/AppendCommand.java)、[InsertCommand](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/command/InsertCommand.java)、[DeleteCommand](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/command/DeleteCommand.java)、[ReplaceCommand](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/command/ReplaceCommand.java) |
| **Invoker**（调用者/调度器） | [CommandManager](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/command/CommandManager.java) |
| **Receiver**（接收者） | [IEditor](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/editor/IEditor.java) |

**核心设计：**

**CommandManager** 维护两个栈：
- `undoStack`：撤销栈，存储已执行的命令
- `redoStack`：重做栈，存储已撤销的命令

```
执行新命令：
  cmd.execute() -> undoStack.push(cmd) -> redoStack.clear()

撤销：
  cmd = undoStack.pop() -> cmd.undo() -> redoStack.push(cmd)

重做：
  cmd = redoStack.pop() -> cmd.execute() -> undoStack.push(cmd)
```

**关键保证：** 执行新命令时**强制清空重做栈**，确保命令历史的一致性。

**宏命令示例 — ReplaceCommand：**

[ReplaceCommand](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/command/ReplaceCommand.java) 是**组合命令（宏命令）**的典型实现，内部组合了 `DeleteCommand` 和 `InsertCommand`：

```java
public class ReplaceCommand implements ICommand {
    private final DeleteCommand deleteCommand;
    private final InsertCommand insertCommand;

    public void execute() {
        deleteCommand.execute();   // 先删除
        insertCommand.execute();   // 后插入
    }

    public void undo() {
        insertCommand.undo();      // 逆序：先撤销插入
        deleteCommand.undo();      // 再撤销删除
    }
}
```

撤销时严格按照**逆序**执行，体现了宏命令的**事务性**语义。

**命令的"快照"机制：**

[DeleteCommand](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/command/DeleteCommand.java) 在执行 `execute()` 时保存被删除的文本到 `deletedText` 字段，撤销时利用该字段恢复原文。这是 Command 模式中保存"状态快照"的典型手法。

### 3.2 Decorator 模式（装饰器模式）

**模式目的：** 动态地为对象添加额外的职责，比继承更灵活。

**参与角色：**

| 角色 | 对应类 |
|------|--------|
| **Component**（抽象组件） | [IEditor](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/editor/IEditor.java) |
| **ConcreteComponent**（具体组件） | [TextEditor](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/editor/TextEditor.java) |
| **Decorator**（抽象装饰器） | [EditorDecorator](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/editor/EditorDecorator.java) |
| **ConcreteDecorator**（具体装饰器） | [SpellCheckDecorator](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/editor/SpellCheckDecorator.java) |

**核心设计：**

[EditorDecorator](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/editor/EditorDecorator.java) 持有 `IEditor delegate` 引用，所有方法默认委托给 delegate：

```java
public abstract class EditorDecorator implements IEditor {
    protected IEditor delegate;

    @Override
    public void append(String text) { delegate.append(text); }
    @Override
    public void insert(int line, int col, String text) { delegate.insert(line, col, text); }
    // ... 其他方法同理
}
```

[SpellCheckDecorator](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/editor/SpellCheckDecorator.java) 继承 `EditorDecorator`，**仅重写** `append` 和 `insert` 方法，在原有逻辑基础上增加拼写检查：

```java
@Override
public void append(String text) {
    delegate.append(text);       // 1. 先执行原始逻辑
    checkSpelling(text);         // 2. 再增加额外行为
}
```

**拼写检查逻辑：** 使用正则表达式 `(?i)[bcdfghjklmnpqrstvwxyz]{3,}` 检测连续 3 个及以上辅音字母，匹配时仅打印 `[Warning]`，不阻断业务流程。

**动态装饰：** 在 CLI 中，`spell-check` 命令通过反射将 `activeEditor` 替换为 `new SpellCheckDecorator(activeEditor)`，实现运行时动态装饰。

### 3.3 Adapter 模式（适配器模式）

**模式目的：** 将一个类的接口转换为客户端期望的另一个接口，使原本不兼容的类可以协同工作。

**参与角色：**

| 角色 | 对应类 |
|------|--------|
| **Target**（目标接口） | [ITreeNode](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/cli/ITreeNode.java) |
| **Adaptee**（被适配者） | [IFileSystem](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/core/IFileSystem.java) + 路径字符串 |
| **Adapter**（适配器） | [FileNodeAdapter](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/cli/FileNodeAdapter.java) |

**核心设计：**

[FileNodeAdapter](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/cli/FileNodeAdapter.java) 将**扁平的** `IFileSystem.listDirectory(path)` 接口适配为**递归的** `ITreeNode` 树形结构：

```java
public class FileNodeAdapter implements ITreeNode {
    private final IFileSystem fs;    // 持有文件系统引用
    private final String path;       // 当前节点路径
    private final String name;       // 节点名称（路径最后一段）

    public List<ITreeNode> getChildren() {
        // 1. 调用 fs.listDirectory(path) 获取直接子条目
        // 2. 对每个子条目递归创建 FileNodeAdapter
        // 3. 返回子节点列表
    }
}
```

**具体示例：**
- 文件系统路径 `a/b/c/file.txt`
- `FileNodeAdapter(fs, "a")` → `getChildren()` 返回 `[b]`
- `FileNodeAdapter(fs, "a/b")` → `getChildren()` 返回 `[c]`
- `FileNodeAdapter(fs, "a/b/c")` → `getChildren()` 返回 `[file.txt]`

配合 [TreePrinter](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/cli/TreePrinter.java) 使用 `├──`、`└──`、`│` 等符号递归渲染可视化的目录树。

### 3.4 Observer 模式（观察者模式）

**模式目的：** 定义一对多的依赖关系，当一个对象状态变化时，所有依赖它的对象都得到通知并自动更新。

**参与角色：**

| 角色 | 对应类 |
|------|--------|
| **Subject**（主题/目标） | [ISubject](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/log/ISubject.java) — 接口；[Workspace](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/workspace/Workspace.java) — 具体实现 |
| **Observer**（观察者） | [ICommandObserver](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/log/ICommandObserver.java) — 接口；[FileLogger](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/log/FileLogger.java) — 具体实现 |

**核心设计：**

[Workspace](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/workspace/Workspace.java) 实现 `ISubject` 接口，维护观察者列表，在 [executeEditorCommand](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/workspace/Workspace.java#L91-L95) 方法中触发通知：

```java
public void executeEditorCommand(ICommand cmd) {
    cmdManager.executeCommand(cmd, logMsg -> notifyObservers(logMsg));
}
```

[FileLogger](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/log/FileLogger.java) 实现 `ICommandObserver`，在 `onCommandExecuted` 中将日志追加到文件：

```java
public void onCommandExecuted(String logMessage) {
    // 第一条日志先写入 "session start at yyyyMMdd HH:mm:ss"
    // 随后每条命令执行都写入一条日志记录
    fs.appendLine(logFilePath, logMessage);
}
```

**关键解耦：**
- Workspace 不直接调用 FileLogger，而是调用 `notifyObservers()`
- FileLogger 不关心命令细节，只接收格式化好的日志字符串
- 观察者可以**动态注册/移除**（`log-on`/`log-off` 命令）

**日志文件内容示例：**
```
session start at 20251024 09:41:40
20251024 09:41:40 append "# log"
20251024 09:41:50 append "Hello World"
```

### 3.5 Memento 模式（备忘录模式）

**模式目的：** 在不破坏封装性的前提下，捕获并外部化对象的内部状态，以便之后可以将对象恢复到之前的状态。

**参与角色：**

| 角色 | 对应类 |
|------|--------|
| **Originator**（原发器） | [Workspace](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/workspace/Workspace.java) |
| **Memento**（备忘录） | [WorkspaceMemento](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/workspace/WorkspaceMemento.java) |

**核心设计：**

[Workspace](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/workspace/Workspace.java) 提供 `saveState()` 和 `restoreState()` 两个方法：

```java
public WorkspaceMemento saveState() {
    // 1. 收集所有打开的文件列表
    // 2. 收集各文件的修改状态
    // 3. 记录当前活动文件路径
    // 4. 将这些状态封装成 WorkspaceMemento 返回
}

public void restoreState(WorkspaceMemento memento) {
    // 1. 恢复活动文件路径
    // 2. 恢复各文件的修改状态
}
```

[WorkspaceMemento](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/workspace/WorkspaceMemento.java) 是一个纯数据类（POJO），包含三个字段：
- `openedFiles` — 已打开的文件路径列表
- `activeFile` — 当前活动文件路径
- `modifiedStates` — 各文件的修改状态映射表

**使用场景（exit 命令）：**
1. 执行 `exit` 时，调用 `workspace.saveState()` 获取当前状态快照
2. 逐一检查未保存的文件，提示用户保存
3. 将工作区状态写入 `.workspace.conf` 配置文件
4. 下次启动时可读取该配置文件恢复工作区状态

### 3.7 Iterator 模式（迭代器模式）

**模式目的：** 提供一种方法顺序访问一个聚合对象中的各个元素，而不暴露其内部表示。

**参与角色：**

**核心设计：**

[LineIterator](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/editor/TextEditor.java#L217-L236) 是 `TextEditor` 的私有内部类，封装了行范围迭代逻辑：

```java
private class LineIterator implements Iterator<String> {
    private int currentIndex;    // 当前索引
    private final int endLine;   // 结束行

    public boolean hasNext() { return currentIndex <= endLine; }

    public String next() {
        if (!hasNext()) throw new EditorException("迭代器越界");
        return lines.get(currentIndex++);
    }
}
```

**设计优势：**
- 对外部完全隐藏 `List<String>` 的内部实现
- 客户端代码只需要通过 `Iterator<String>` 接口遍历，无需关心底层数据结构
- 支持指定 `startLine:endLine` 范围的部分遍历

**核心设计：**

[Workspace](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/workspace/Workspace.java) 作为系统的**统一门面**，向 CLI 层提供以下高层接口：

**Facade 的作用：**

### 4.3 异常安全屏障

CLIApplication 的主循环采用**最高安全级别**的三层防护：

```java
try {
    // 1. 业务逻辑
} catch (EditorException e) {
    // 2. 已知业务异常 → 打印友好提示
    System.out.println(e.getMessage());
} catch (Exception e) {
    // 3. 未知系统异常 → 打印带 [SysError] 前缀
    System.out.println("[SysError] " + e.getMessage());
}
```

任何异常都不会导致程序退出，保证 CLI 的持续可用性。

## 六、关键设计决策

### 6.1 行号/列号从 1 开始

所有行号和列号均从 **1** 开始计数，与用户直觉一致，降低使用成本。这也是主流文本编辑器（Vim、VS Code 等）的实践。

### 6.2 RuntimeException 体系

[EditorException](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/core/EditorException.java) 继承自 `RuntimeException`（非受检异常），原因：
- 避免在 CLI 命令分发链中强制 try-catch 或 throws 声明
- 简化代码，让开发者专注于业务逻辑

### 6.3 TDD 先导的开发方式

项目遵循**测试驱动开发**：
1. 先定义接口（如 `IFileSystem`、`IEditor`）
2. 提供 Mock 实现（如 `MockFileSystem`）
3. 编写测试用例验证接口行为
4. 提供真实实现（如 `LocalFileSystem`）
5. 在真实环境中运行验证

### 6.4 装饰器的动态应用

`SpellCheckDecorator` 在 `spell-check` 命令触发时，通过**反射**将 `Workspace.editors` 中的 `IEditor` 对象替换为装饰后的对象，实现运行时行为扩展，无需重新加载文件。

### 6.5 迭代器的"宽容模式"

[TextEditor.getLineIterator](file:///d:/Erdong/大二/OOT/lab1/lab2/src/main/java/editor/TextEditor.java#L189-L203) 采用了**自动修正**的宽容策略：
- 空文件返回空迭代器而非抛出异常
- 越界的行号自动 `Math.max`/`Math.min` 修正到合法范围
- 仅当修正后起始行 > 结束行时才视为真正的逻辑错误
